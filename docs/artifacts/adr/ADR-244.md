## Context

AgentX already has strong workflow primitives: canonical checkpoints, execution plans, progress logs, loop state, durable review artifacts, and extension guidance. However, the current `Work` checkpoint does not yet carry a first-class bounded contract model for complex slices.

**Requirements:**
- Add a durable bounded contract for complex work slices.
- Keep the existing checkpoint vocabulary intact.
- Support evaluator participation during `Work` when complexity warrants it.
- Strengthen runtime evidence expectations across docs, CLI/runtime, and extension surfaces.
- Preserve staged rollout and avoid overstating enforcement before runtime support exists.

**Constraints:**
- The repo already uses `Brainstorm`, `Plan`, `Work`, `Review`, `Compound Capture`, and `Done` as canonical checkpoints.
- Existing docs explicitly warn against introducing a second workflow state machine.
- Current runtime enforcement is partial and must remain honest.
- The design must stay repo-native and artifact-first.

**Background:**
- Internal repo evidence:
  - `docs/WORKFLOW.md` defines deterministic checkpoint resolution and durable-evidence requirements. Confidence: HIGH.
  - `vscode-extension/src/utils/workflowGuidance.ts` already resolves workflow state from durable files rather than transcript-only context. Confidence: HIGH.
  - `vscode-extension/src/utils/harnessState.ts` already provides file-backed harness-thread state. Confidence: HIGH.
  - `.agentx/agentic-runner.ps1` already supports loop state, compaction, clarification, and resumable execution. Confidence: HIGH.
- External research:
  - Anthropic's harness-design guidance emphasizes planner/implementer/evaluator separation, bounded contracts, real-surface validation, durable artifacts, and removal of non-load-bearing scaffolding. Confidence: HIGH.

**Failure modes researched:**
- Keeping only broad phase transitions makes complex work harder to bound and resume. Confidence: HIGH.
- Introducing a second lifecycle vocabulary would confuse operators and split durable state. Confidence: HIGH.
- Mandatory evaluator involvement for every task would create unnecessary friction and cost. Confidence: HIGH.
- Treating the initiative as UI-only or docs-only would fail to solve the runtime contract gap. Confidence: HIGH.

**Security and long-term viability assessment:**
- Confidence: HIGH - artifact-first contracts align with AgentX's existing repository-local operating model.
- Confidence: HIGH - adaptive evaluator use is more sustainable than mandatory evaluator loops on every task.
- Confidence: MEDIUM - reset/compaction policy may need tuning by model/provider over time, so the design should remain policy-driven rather than hardcoded to one model family.

---

## Decision

We will extend AgentX with contract-driven work slices nested inside the existing `Work` checkpoint, supported by adaptive evaluator participation and stronger runtime evidence expectations.

**Key architectural choices:**
- Reuse the existing checkpoint vocabulary rather than introducing a parallel harness lifecycle.
- Represent bounded complex work through durable slice-contract artifacts or equivalent durable state.
- Add evaluator participation selectively during `Work` based on complexity, risk, and surface sensitivity.
- Separate implementation evidence, verification evidence, and runtime evidence in the architecture.
- Treat context compaction and clean-slate resets as explicit policy decisions, not accidental side effects.

---

## Options Considered

### Option 1: Keep The Existing Checkpoint Model With No Slice Contract Layer

**Description:**
Continue relying on plans, progress logs, and final review without adding a bounded contract abstraction inside `Work`.

**Pros:**
- Lowest implementation effort
- No new artifact family required
- Preserves all current surfaces unchanged

**Cons:**
- Complex work remains harder to bound and resume
- Evaluator feedback stays too concentrated at final review
- Runtime evidence requirements remain underspecified during implementation

**Effort**: S
**Risk**: High
**Confidence**: LOW

---

### Option 2: Add A New Parallel Harness Lifecycle Separate From Checkpoints

**Description:**
Introduce a second state machine for contract, evaluator, and reset behavior beside the current checkpoint model.

**Pros:**
- Could isolate harness behavior very explicitly
- Gives maximum freedom for a dedicated harness vocabulary

**Cons:**
- Splits operator mental model
- Risks contradictory durable state across checkpoint and harness layers
- Violates the current repo direction toward one shared lifecycle language

**Effort**: M
**Risk**: High
**Confidence**: LOW

---

### Option 3: Extend `Work` With Contract-Driven Slices And Adaptive Evaluator Loops

**Description:**
Keep the existing checkpoint system and add bounded work contracts, evaluator findings, and explicit evidence expectations inside `Work`.

**Pros:**
- Builds on existing AgentX vocabulary and artifacts
- Improves complex-work discipline without rewriting the lifecycle
- Keeps evaluator use selective and pragmatic
- Aligns with repo-native artifact and review patterns

**Cons:**
- Requires new artifact/state semantics and targeted runtime changes
- Needs careful rollout to avoid documentation/runtime drift

**Effort**: M
**Risk**: Low
**Confidence**: HIGH

---

### Option 4: Make Evaluator Participation Mandatory For All Work

**Description:**
Require every task, including small bugs and docs changes, to pass through a formal evaluator loop before work can progress.

**Pros:**
- Maximum consistency of evaluation semantics
- Stronger formalism for proof and feedback

**Cons:**
- Too heavy for routine work
- Increases latency and cost on low-risk tasks
- Likely to create bypass behavior rather than disciplined adoption

**Effort**: M
**Risk**: Medium
**Confidence**: MEDIUM

---

## Rationale

We chose **Option 3** because it adds the highest-value harness improvements while preserving AgentX's strongest existing design choice: one durable workflow vocabulary grounded in repo-local evidence.

**Key decision factors:**
- Preserve checkpoint continuity
- Improve bounded execution for complex tasks
- Bring evaluator feedback earlier without burdening every task
- Keep the architecture adaptable as model capability evolves

1. **Checkpoint continuity matters more than novelty**: AgentX already has a usable lifecycle contract, so the best architecture is to strengthen `Work`, not replace the lifecycle. Confidence: HIGH.
2. **Bounded contracts are the missing operational unit**: Plans and reviews are already durable, but the slice between them is not yet explicit enough. Confidence: HIGH.
3. **Adaptive evaluator use matches the repo's pragmatic stance**: Evaluator loops should remain load-bearing only where complexity justifies them. Confidence: HIGH.
4. **Explicit context policy is necessary for long-running work**: Reset vs compaction needs to be intentional and evidence-aware rather than implicit runtime behavior. Confidence: MEDIUM.

---

## Consequences

### Positive
- Complex work gains a clearer bounded execution unit.
- Evaluator findings can inform implementation before final review.
- Runtime evidence expectations become easier to explain and validate.
- Future model/runtime changes can simplify the harness without rewriting the workflow system.

### Negative
- AgentX needs new artifact/state semantics and runtime wiring.
- Docs, CLI/runtime, and extension guidance must be kept in sync through rollout.
- Pilot and pruning work becomes part of the initiative, not an optional afterthought.

### Neutral
- Existing plans, reviews, and checkpoint logic remain valid but will become more explicit around `Work`.
- Not every task will use the full slice/evaluator pattern.

## Implementation

**Detailed technical specification**: [SPEC-244.md](../specs/SPEC-244.md)

**High-level implementation plan:**
1. Formalize contract artifacts and workflow semantics.
2. Add runtime support for contract lifecycle, evaluator findings, and evidence summaries.
3. Add extension/operator visibility for active slice state and next steps.
4. Pilot on selected complex work and prune non-load-bearing scaffolding.

**Key milestones:**
- Phase 1: PRD, ADR, spec, and issue hierarchy complete.
- Phase 2: Workflow/docs/template foundation complete.
- Phase 3: Runtime and extension implementation complete for pilot surfaces.
- Phase 4: Pilot review and pruning guidance complete.
