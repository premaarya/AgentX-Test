## Context

AgentX already stores runtime observations and memory artifacts, but story #162 requires a narrower contract: a curated learnings schema that supports metadata filtering and deterministic ranking for planning and review entry points. Raw session observations are useful for local recall, but they do not by themselves express whether a conclusion is reusable, validated, relevant to a workflow phase, or suitable for retrieval outside the originating session. [Confidence: HIGH]

The workflow contract in story #163 defines when compound capture occurs. This story defines what a durable learning looks like once capture is required or accepted, and how retrieval should rank those learnings before any later model synthesis. [Confidence: HIGH]

### AI-First Assessment

Could this problem be solved better by GenAI or agentic AI alone? No. Ranking and narrowing should begin with deterministic metadata and explicit evidence signals. AI can summarize retrieved learnings later, but the retrieval contract must remain inspectable, low-noise, and reproducible from repo-local data. [Confidence: HIGH]

### Requirements

- Define metadata required for narrowing and ranking reusable learnings. [Confidence: HIGH]
- Distinguish durable learnings from runtime-only session observations. [Confidence: HIGH]
- Support category narrowing, keyword overlap, and relevance ranking. [Confidence: HIGH]
- Align with current AgentX memory, docs, and issue patterns. [Confidence: HIGH]
- Avoid requiring embeddings or an external search service for phase 1. [Confidence: HIGH]

### Constraints

- The design must fit both local and GitHub-backed workflows. [Confidence: HIGH]
- Ranking must remain deterministic enough for testing and review. [Confidence: HIGH]
- The schema must not force every runtime observation to become a durable learning. [Confidence: HIGH]
- The contract must stay compatible with future richer retrieval without requiring a rewrite. [Confidence: HIGH]

### Known Failure Modes And Anti-Patterns

- Treating every observation as a reusable learning creates noise.
- Ranking purely by free-text search misses workflow relevance and evidence quality.
- A learnings record with no source or evidence links becomes hard to trust.
- Storing only one generic category makes narrowing too weak for planning and review.
- Coupling ranking to embeddings too early hides why an item ranked highly. [Confidence: HIGH]

## Decision

We will introduce a curated learnings schema separate from runtime-only observation records and rank those learnings using deterministic metadata-first scoring.

**Key architectural choices:**

- Separate `LearningRecord` from raw runtime observation records. [Confidence: HIGH]
- Require each durable learning to carry source references, lifecycle context, evidence strength, and applicability metadata. [Confidence: HIGH]
- Rank learnings using deterministic factors: category match, phase match, keyword overlap, evidence strength, validation state, recency, and reuse signal. [Confidence: HIGH]
- Make metadata narrowing mandatory before any optional summarization step. [Confidence: HIGH]
- Preserve compatibility with later semantic retrieval by keeping the schema explicit and extensible. [Confidence: HIGH]

## Options Considered

### Option 1: Reuse The Existing Observation Schema Unchanged

**Description:** Treat current memory observations as the durable learnings format and extend retrieval only with search tweaks.

**Pros:**
- Lowest immediate effort
- Reuses current storage concepts directly
- No new artifact type required

**Cons:**
- Raw observations are too noisy for planning and review entry points
- Does not distinguish validated learnings from session-only notes
- Weak support for lifecycle and evidence-aware ranking

**Effort**: S
**Risk**: High

---

### Option 2: Create A Separate Curated Learnings Schema With Deterministic Ranking

**Description:** Introduce a durable learning record with explicit metadata and a deterministic ranking contract over those records.

**Pros:**
- Clean separation between runtime observations and reusable knowledge
- Supports explainable narrowing and ranking
- Fits the compound-capture workflow from story #163
- Keeps phase-1 retrieval low-noise and testable

**Cons:**
- Adds a second data shape that must stay mapped to source artifacts
- Requires explicit capture discipline to populate useful records
- Needs follow-on implementation work in retrieval entry points

**Effort**: M
**Risk**: Medium

---

### Option 3: Use Vector-Or Semantic-Only Retrieval For Learnings

**Description:** Store learning text and rely primarily on semantic embeddings or vector search to rank relevance.

**Pros:**
- Stronger similarity matching for fuzzy queries
- Potentially powerful for later scale
- Can support richer paraphrase queries

**Cons:**
- Harder to explain and validate in phase 1
- Adds provider, index, or embedding concerns too early
- Weakens deterministic ranking guarantees for current acceptance criteria

**Effort**: L
**Risk**: High

---

### Option 4: Add Metadata Tags But Rank Mostly By Free-Text Search

**Description:** Keep schema changes light and use keyword matching as the dominant ranking signal.

**Pros:**
- Simple to implement
- Better than unstructured browsing
- No new advanced retrieval logic required

**Cons:**
- Underuses workflow and evidence metadata
- Performs poorly when phrasing differs across issues
- Does not deliver the intended category and lifecycle sensitivity

**Effort**: S
**Risk**: Medium

## Evaluation

| Criteria | Option 1 | Option 2 | Option 3 | Option 4 |
|----------|----------|----------|----------|----------|
| Distinguishes durable knowledge from runtime notes | Low | High | Medium | Medium |
| Deterministic and explainable ranking | Medium | High | Low | Medium |
| Supports category and phase narrowing | Low | High | Medium | Medium |
| Works with current AgentX storage patterns | High | High | Low | High |
| Delivery risk for current initiative | Medium | Medium | High | Medium |

**Result:** Option 2 is preferred because it gives AgentX a curated, explainable retrieval surface without overcommitting to semantic infrastructure too early. [Confidence: HIGH]

---

## Rationale

We chose **Option 2: Separate Curated Learnings Schema With Deterministic Ranking** because it best aligns with the PRD and the workflow contract.

1. **Planning and review need higher-signal inputs than raw memory**: durable learnings should represent reviewed, reusable outcomes rather than every observed session detail. [Confidence: HIGH]
2. **Ranking must be inspectable**: category, phase, evidence, and validation state should explain why one learning ranks above another. [Confidence: HIGH]
3. **Future semantic retrieval should be additive**: starting with strong metadata keeps phase 1 simple while preserving a clean upgrade path later. [Confidence: HIGH]

**Key decision factors:**

- Low-noise retrieval for planning and review
- Alignment with compound-capture lifecycle rules
- Reuse of current AgentX memory and documentation patterns

---

## Consequences

### Positive

- Durable learnings become more trustworthy and easier to filter.
- Retrieval results can explain why each item surfaced.
- Runtime observations remain useful without polluting the durable learnings set.
- Later retrieval wiring has a stable contract to target. [Confidence: HIGH]

### Negative

- Agents and operators must decide which captured outcomes merit a curated learning record.
- Two related knowledge shapes now exist: observations and curated learnings.
- Follow-on implementation work is required to map current memory artifacts into the new contract. [Confidence: MEDIUM]

### Neutral

- Phase 1 ranking remains metadata-first and does not require semantic search.
- This ADR does not define the user interface that will display retrieved learnings.
