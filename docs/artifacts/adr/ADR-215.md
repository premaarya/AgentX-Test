## Context

AgentX already has meaningful workflow primitives for issue-first planning, architecture artifacts, execution plans, review artifacts, learnings capture, durable review findings, and packaged compatibility assets. The product problem is not absence of capability. The problem is that those capabilities are experienced as adjacent features rather than one coherent operating system for delivery. [Confidence: HIGH]

PRD-215 defines three primary goals: make the guided operating loop explicit, create a phased backlog for bounded orchestration and portability, and introduce rollout discipline before broader workflow expansion. The execution plan for the initiative further decomposes this into six workstreams: workflow cohesion, bounded parallel delivery, task bundle operations, skill packaging, host portability, and governance. [Confidence: HIGH]

The architectural decision for this epic must answer four questions:

1. What is the stable control plane for brainstorm, plan, work, review, and compound capture?
2. How should bounded parallel work fit into the current AgentX lifecycle without creating a second state model?
3. How should task decomposition and portability build on existing repo-local artifacts instead of introducing parallel trackers or hand-maintained host forks?
4. How should AI participate in the workflow without replacing deterministic workflow control? [Confidence: HIGH]

### AI-First Assessment

Could this initiative be solved better by an AI-only orchestration layer? No. The workflow checkpoint model, rollout scorecards, decomposition eligibility rules, and portability boundaries must remain deterministic, inspectable, and repo-native. AI is still valuable for checkpoint guidance, context narrowing, recommendation generation, review assistance, and future orchestration support, but the control plane itself should stay mechanical and artifact-based. [Confidence: HIGH]

### Requirements

- Preserve the current AgentX issue hierarchy and workflow states. [Confidence: HIGH]
- Reuse existing artifact families: PRDs, ADRs, specs, execution plans, progress logs, reviews, learnings, and review findings. [Confidence: HIGH]
- Make phase-1 workflow cohesion and rollout discipline additive and low-risk. [Confidence: HIGH]
- Allow later task bundles, bounded parallel delivery, and pack generation to extend the same control plane. [Confidence: HIGH]
- Keep product identity stable across portability work. [Confidence: HIGH]

### Constraints

- No second backlog or sidecar task system becomes the source of truth. [Confidence: HIGH]
- No host-specific workflow contract replaces the repo-local contract. [Confidence: HIGH]
- Portability must come from canonical repo assets, not divergent hand-maintained copies. [Confidence: HIGH]
- Workflow guidance must remain understandable in both GitHub-backed and local artifact-centric modes. [Confidence: HIGH]

### Known Failure Modes And Anti-Patterns

- Operators cannot tell which checkpoint they are in and must infer next steps manually.
- Parallel work starts without eligibility rules, producing merge debt and review ambiguity.
- Decomposed tasks become a second tracker instead of promoting into normal AgentX issues.
- Portability work drifts because generated surfaces are not derived from canonical assets.
- AI suggestions overrule deterministic workflow evidence instead of complementing it. [Confidence: HIGH]

## Decision

We will implement epic #215 as a phased workflow-expansion architecture with three layers:

1. **Workflow Control Plane**: an artifact-centric checkpoint model for brainstorm, plan, work, review, and compound capture. [Confidence: HIGH]
2. **Bounded Orchestration Layer**: later-phase task-bundle and parallel-delivery capabilities that extend the control plane without changing the underlying workflow states. [Confidence: HIGH]
3. **Canonical Portability Layer**: generated host-facing packs and compatibility outputs derived from repo-native source artifacts. [Confidence: HIGH]

Key architectural choices:

- Treat checkpoint state as a function of existing artifacts and workflow evidence, not transient chat context. [Confidence: HIGH]
- Make phase-1 operator guidance advisory-first, using deterministic evidence before AI synthesis. [Confidence: HIGH]
- Use task bundles as a later decomposition mechanism that remains subordinate to the standard epic -> feature -> story model. [Confidence: HIGH]
- Treat bounded parallel delivery as an opt-in mode gated by eligibility rules and reconciliation review, not a default execution behavior. [Confidence: HIGH]
- Generate portability outputs from canonical commands, docs, instructions, and pack assets so product identity and workflow contracts remain stable. [Confidence: HIGH]

## Options Considered

### Option 1: Keep The Existing Workflow Surfaces And Add Incremental Commands Only

**Description:** Preserve the current product model and add new commands or guides where needed without a unifying control-plane architecture.

**Pros:**
- Lowest immediate change cost
- Minimal architectural disruption
- Faster to ship isolated improvements

**Cons:**
- Does not solve the core cohesion problem
- Makes later task-bundle and portability work more fragmented
- Keeps operator experience dependent on tribal knowledge

**Effort**: S
**Risk**: High

---

### Option 2: Replace The Current Workflow Model With A New Orchestration-First System

**Description:** Rebuild AgentX around a new execution-first orchestration model that treats decomposition, parallel work, and portability as primary concepts.

**Pros:**
- Clean conceptual model from day one
- Strong room for advanced orchestration
- Could unify future work under one redesigned system

**Cons:**
- Disruptive to current docs, commands, tests, and operator expectations
- High migration cost
- High risk of over-design before phase-1 value is proven

**Effort**: L
**Risk**: High

---

### Option 3: Add A Phased Control Plane Over Existing Artifacts And Extend It Incrementally

**Description:** Introduce a stable checkpoint-based workflow control plane over the current AgentX lifecycle, then layer decomposition, bounded orchestration, and portability generation on top.

**Pros:**
- Reuses current artifact families and workflow semantics
- Delivers immediate phase-1 value through better operator guidance
- Keeps later workstreams compatible with current docs and runtime surfaces
- Reduces risk by separating control-plane clarity from advanced automation

**Cons:**
- Requires careful contract design to avoid ambiguity between current and expanded surfaces
- Some benefits arrive only in later phases
- Adds architecture and documentation overhead upfront

**Effort**: M
**Risk**: Medium

---

### Option 4: Build Portability First And Let Workflow Cohesion Follow

**Description:** Prioritize generated pack and host-surface parity first, then revisit workflow cohesion and bounded orchestration later.

**Pros:**
- Tangible compatibility outputs early
- Easier to market as broad host support

**Cons:**
- Optimizes distribution before fixing product workflow clarity
- Risks exporting fragmented user experience to more hosts
- Makes future generated surfaces harder to standardize

**Effort**: M
**Risk**: High

## Evaluation

| Criteria | Option 1 | Option 2 | Option 3 | Option 4 |
|----------|----------|----------|----------|----------|
| Improves workflow cohesion | Low | Medium | High | Low |
| Reuses current AgentX artifacts and states | High | Low | High | Medium |
| Supports bounded later expansion | Low | High | High | Medium |
| Delivery risk | Medium | High | Medium | High |
| Portability stability | Low | Medium | High | Medium |
| Time-to-value for phase 1 | Medium | Low | High | Low |

**Result:** Option 3 is preferred because it addresses the real workflow problem first while preserving a safe path toward later orchestration and portability layers. [Confidence: HIGH]

---

## Rationale

We chose **Option 3: Add A Phased Control Plane Over Existing Artifacts And Extend It Incrementally** because it is the only option that aligns product value, delivery risk, and current repository reality.

1. **The core problem is control-plane clarity, not missing raw capability.** AgentX already has learnings, reviews, execution plans, packs, and strong workflow guidance. The missing architecture is the model that connects them. [Confidence: HIGH]
2. **Phase-1 value depends on better guidance, not heavier automation.** Operators need explicit checkpoint language, entry points, and rollout discipline before they need parallel work or portability expansion. [Confidence: HIGH]
3. **Later workstreams need a stable substrate.** Task bundles, bounded parallel delivery, skill packaging, and generated packs become safer once the workflow control plane is explicit and durable. [Confidence: HIGH]
4. **AI should assist the workflow, not define it.** Deterministic artifact evidence should anchor recommendations so the workflow remains explainable and consistent across surfaces. [Confidence: HIGH]

## Consequences

### Positive

- AgentX gets a durable control-plane architecture for workflow expansion. [Confidence: HIGH]
- Phase-1 can ship immediate operator value without waiting for advanced orchestration features. [Confidence: HIGH]
- Later decomposition and portability work can reuse the same artifact-centric model. [Confidence: HIGH]
- Generated host surfaces can remain aligned with the canonical repo contract. [Confidence: HIGH]

### Negative

- The initiative adds documentation and architecture overhead before full implementation value is visible. [Confidence: MEDIUM]
- Some operators may see temporary overlap between old broad workflow selection and new checkpoint guidance until surfaces are fully aligned. [Confidence: MEDIUM]
- Later task-bundle and portability work will still require additional architectural and implementation slices. [Confidence: HIGH]

### Neutral

- This ADR does not define the final task-bundle schema, reconciliation algorithm, or portability generation mechanics.
- Advisory-first guidance remains the rollout mode until future slices prove stronger automation readiness.

## Implementation

**Detailed technical specification**: [SPEC-215.md](../specs/SPEC-215.md)

**High-level implementation plan:**

1. Deliver the workflow control plane through checkpoint contracts, entry points, and next-step guidance.
2. Add rollout scorecards and pilot exit criteria before broader feature waves.
3. Introduce task bundles as the decomposition substrate for later bounded parallel delivery.
4. Add bounded orchestration rules, isolated task units, and reconciliation review.
5. Extend skill packaging metadata and authoring flows.
6. Add canonical portability generation and compatibility validation.

**Key milestones:**

- Phase 1: Workflow cohesion and rollout governance
- Phase 2: Task-bundle contract and bounded parallel-delivery pilot
- Phase 3: Skill packaging uplift and portability-generation contract
