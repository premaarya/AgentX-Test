## Context

AgentX already has strong workflow primitives, but the transition from finished work to reusable institutional knowledge is still implied rather than contractual. The repo contains issue-first routing, execution-plan requirements for complex work, review artifacts, persistent memory notes, and harness evaluation, but there is no single workflow contract that says which artifact must exist at each phase, what qualifies as an acceptable handoff, and when knowledge capture is mandatory versus optional. [Confidence: HIGH]

This gap matters because story #163 is the dependency that turns the PRD intent into implementation-ready rules. Without a lifecycle contract, later work on learnings schema, retrieval ranking, review integration, and durable finding promotion will drift into separate features instead of one coherent loop. [Confidence: HIGH]

### AI-First Assessment

Could this problem be solved better by GenAI or agentic AI alone? No. The workflow contract itself must be deterministic, explainable, and repo-native. AI can assist later with summarization, retrieval, or advisory review, but the phase contract and artifact gates need to remain mechanical so they work across local mode, GitHub mode, CLI, and VS Code surfaces. [Confidence: HIGH]

### Requirements

- Define the required artifacts for planning, architecture, execution, review, and compound capture. [Confidence: HIGH]
- Define entry and exit criteria for each phase so handoffs are inspectable. [Confidence: HIGH]
- Distinguish mandatory versus optional knowledge-capture scenarios. [Confidence: HIGH]
- Keep the contract host-neutral so it does not depend on one chat client or command surface. [Confidence: HIGH]
- Reuse existing AgentX issue, docs, and memory patterns rather than adding a parallel workflow system. [Confidence: HIGH]

### Constraints

- The contract must remain compatible with both local mode and GitHub-backed mode. [Confidence: HIGH]
- The contract must avoid introducing a second backlog or sidecar state machine. [Confidence: HIGH]
- The contract must work before story #162 finalizes the learnings schema and ranking model. [Confidence: HIGH]
- The deliverable must remain ASCII-only and repo-local. [Confidence: HIGH]

### Known Failure Modes And Anti-Patterns

- Work is marked complete with no durable summary of the solved problem.
- Review findings exist, but no one can tell whether the outcome should produce a reusable learning.
- Artifact expectations vary by execution surface, so operators improvise the phase contract.
- Knowledge capture is attempted too early, before review and evidence stabilize.
- A separate backlog or sidecar tracker duplicates AgentX issue state and fragments ownership. [Confidence: HIGH]

## Decision

We will define an explicit knowledge-compounding workflow contract that overlays the existing AgentX lifecycle with a dedicated compound-capture phase and clear artifact handoff rules.

**Key architectural choices:**

- Model the lifecycle as five phases: plan, design, execute, review, and compound capture. [Confidence: HIGH]
- Treat artifact classes, not tools, as the contract boundary so the workflow stays host-neutral. [Confidence: HIGH]
- Reuse existing AgentX artifact families: issue records, PRD or story context, ADR and spec where required, execution plan, progress log, review artifact, and memory or solution outputs. [Confidence: HIGH]
- Make compound capture conditional on evidence and review outcome instead of unconditional on every task. [Confidence: HIGH]
- Defer detailed learnings fields and ranking behavior to story #162 while fixing the lifecycle rules here. [Confidence: HIGH]

## Options Considered

### Option 1: Keep The Workflow Implicit

**Description:** Continue relying on existing docs and role instructions without a dedicated contract for compound capture.

**Pros:**
- No additional design artifact required
- Lowest short-term effort
- Preserves current flexibility

**Cons:**
- Leaves phase expectations inconsistent
- Makes later learnings retrieval harder to standardize
- Keeps knowledge capture optional by habit rather than by rule

**Effort**: S
**Risk**: High

---

### Option 2: Add A Separate Knowledge Backlog And Capture Workflow

**Description:** Create a parallel workflow with its own backlog objects and artifact chain for reusable learnings.

**Pros:**
- Makes knowledge work highly visible
- Separates capture concerns from normal delivery
- Could support specialized operators later

**Cons:**
- Duplicates issue state and ownership
- Increases workflow overhead for maintainers
- Conflicts with the repo rule against fragmented tracking systems

**Effort**: M
**Risk**: High

---

### Option 3: Add An Explicit Lifecycle Contract Over Existing Artifacts

**Description:** Define phase entry and exit rules, required artifacts, and capture-decision rules using the artifact families AgentX already has.

**Pros:**
- Aligns with current AgentX workflow model
- Works across local and GitHub-backed modes
- Creates a stable foundation for stories #162, #166, and #169
- Avoids a second source of truth

**Cons:**
- Requires precise phase definitions to avoid ambiguity
- Depends on later stories to supply schema and retrieval behavior
- Adds some documentation and validation surface area

**Effort**: M
**Risk**: Medium

---

### Option 4: Couple The Contract To Specific Host Commands Or Chat Surfaces

**Description:** Express the lifecycle through tool-specific or host-specific steps tied to a single interaction surface.

**Pros:**
- Easy to demonstrate in one environment
- Can feel concrete to operators in that surface

**Cons:**
- Violates the acceptance criterion for cross-surface compatibility
- Becomes brittle as AgentX evolves across CLI, extension, and GitHub workflows
- Encodes implementation detail where the contract should describe intent

**Effort**: S
**Risk**: High

## Evaluation

| Criteria | Option 1 | Option 2 | Option 3 | Option 4 |
|----------|----------|----------|----------|----------|
| Reuses current AgentX workflow | Medium | Low | High | Medium |
| Supports local and GitHub modes | Medium | Low | High | Low |
| Enables later learnings retrieval | Low | Medium | High | Medium |
| Avoids workflow fragmentation | High | Low | High | Medium |
| Keeps host-specific assumptions out | High | Medium | High | Low |
| Delivery risk | Low | High | Medium | High |

**Result:** Option 3 is preferred because it adds the missing workflow discipline without creating a second state model or binding the design to one host surface. [Confidence: HIGH]

---

## Rationale

We chose **Option 3: Add An Explicit Lifecycle Contract Over Existing Artifacts** because it is the smallest change that closes the real architectural gap.

1. **AgentX already has the right artifact families**: issues, plans, progress, reviews, and memory are present; the missing piece is the contract that connects them. [Confidence: HIGH]
2. **The contract must outlive current tooling details**: if the lifecycle is written in terms of chat commands or one specific surface, later integrations will drift. Artifact-based rules are more durable. [Confidence: HIGH]
3. **Later stories depend on a stable phase model**: story #162 needs a lifecycle destination for learnings records, and stories #166 and #169 need predictable entry points for capture and retrieval. [Confidence: HIGH]

**Key decision factors:**

- Preserve issue-first workflow and current repo conventions
- Make knowledge capture explicit but not universal
- Keep the design implementation-ready for subsequent engineering work

---

## Consequences

### Positive

- Maintainers get a single contract that explains when each artifact is required.
- Later schema and retrieval work can build on a stable lifecycle.
- Review and capture decisions become evidence-based instead of ad hoc.
- The same workflow can be expressed in local mode and GitHub mode. [Confidence: HIGH]

### Negative

- Operators must learn one additional lifecycle phase: compound capture.
- Some current docs and role instructions will need follow-on alignment work.
- Validation will remain partially manual until later automation stories land. [Confidence: MEDIUM]

### Neutral

- This ADR does not itself define the learnings schema or ranking algorithm.
- Existing issue and review artifacts remain the source of truth for delivery state.
