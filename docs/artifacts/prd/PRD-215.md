## 1. Problem Statement

### What problem are we solving?

AgentX already has strong planning, execution, review, learning-capture, and packaging primitives, but those capabilities still feel fragmented in day-to-day use. Operators can reach brainstorm guidance, review augmentation, compound capture, and packaging assets, yet the product does not present them as one coherent operating loop and does not yet provide a clear product path for bounded parallel delivery, task bundles, or generated host portability.

### Why is this important?

- Workflow cohesion increases operator confidence and reduces manual stitching between planning, execution, review, and knowledge capture.
- A phased product backlog makes it possible to add bounded orchestration and portability safely instead of through ad-hoc expansion.
- Stronger rollout discipline keeps new workflow surfaces measurable, reversible, and aligned with the rest of AgentX's product contract.

### What happens if we do not solve this?

AgentX will continue to expose useful capabilities through separate surfaces without a clear operating path, which increases operator friction, weakens discoverability, and slows future work on parallel delivery and portability. The product will keep adding power without enough cohesion.

## 4. Research Summary

### Sources Consulted

- Internal workflow contracts in `docs/WORKFLOW.md` and `AGENTS.md`
- Compound workflow and review guidance in `docs/guides/KNOWLEDGE-REVIEW-WORKFLOWS.md`
- Existing extension workflow and status surfaces in `vscode-extension/src/commands/workflow.ts`, `vscode-extension/src/views/statusTreeProvider.ts`, and `vscode-extension/package.json`
- Packaging and portability scaffolding in `packs/`, install flows, and `.github/schemas/plugin-manifest.schema.json`
- Existing repo decisions and conventions in `memories/decisions.md`, `memories/conventions.md`, and prior initiative artifacts

### Key Findings

1. AgentX already has meaningful native surfaces for brainstorm guidance, review augmentation, compound capture, learning retrieval, and durable follow-up promotion.
2. The largest gap is surface cohesion and sequencing, not missing concepts.
3. Parallelism is already accepted in validation and testing guidance, but not yet formalized as a bounded delivery model.
4. Packaging and portability foundations already exist through packs, schemas, bundled docs, and installers.
5. The safest next step is to unify the current workflow loop and add rollout discipline before deeper orchestration or portability automation.

### Chosen Approach Rationale

Use a phased, additive expansion program. Start by making the guided loop explicit, measurable, and easy to traverse across product surfaces. Then introduce task bundles and bounded parallel delivery. After those contracts stabilize, extend skill packaging and pack-based portability from canonical AgentX assets.

### Rejected Alternatives

- Replace the current AgentX lifecycle with a new workflow model.
  Reason: Too disruptive and unnecessary given the existing workflow and artifact contracts.
- Add a second tracker for decomposition and follow-up work.
  Reason: AgentX already has a durable issue and workflow model.
- Start with portability automation first.
  Reason: Lower user value if the underlying workflow surfaces remain fragmented.

## 5. Requirements

### 5.1 Functional Requirements

#### Must Have (P0)

1. **Guided workflow loop contract**
   - **User Story**: As an AgentX operator, I want brainstorm, plan, work, review, and compound capture to behave like one clear loop so that I know how to move through delivery without guessing.
   - **Acceptance Criteria**:
   - [ ] Each checkpoint has documented entry criteria, exit criteria, and artifact expectations
   - [ ] Shared naming is used across docs, commands, chat, and sidebar surfaces
   - [ ] The model reuses current AgentX workflow states instead of adding a second state machine

2. **Phase-1 operator entry points**
   - **User Story**: As an operator, I want explicit plan-deepening and review-kickoff entry points so that I can move into the next checkpoint with the right context.
   - **Acceptance Criteria**:
   - [ ] At least two operator surfaces expose the new entry points
   - [ ] Entry points reuse current issue, plan, learning, and review context when available
   - [ ] Outputs are resumable and advisory-first

3. **Rollout scorecard and pilot discipline**
   - **User Story**: As a maintainer, I want measurable rollout criteria so that workflow expansion ships in bounded slices with clear success and rollback signals.
   - **Acceptance Criteria**:
   - [ ] Phase-1 slices have success, warning, and rollback thresholds
   - [ ] Pilot order and exit criteria are documented before implementation starts
   - [ ] Release enablement guidance exists for new operator surfaces

#### Should Have (P1)

1. **Context-sensitive next-step recommendations**
   - **User Story**: As an operator, I want the product to recommend the next workflow action from current artifact state so that I can resume work faster.
   - **Acceptance Criteria**:
   - [ ] Work and Status surfaces can show the same recommendation
   - [ ] Recommendations react to existing plan, review, and compound evidence
   - [ ] Guidance is testable and side-effect free

2. **Task bundle product contract**
   - **User Story**: As a maintainer, I want decomposed work to use a consistent bundle model so that later bounded parallel delivery is traceable and recoverable.
   - **Acceptance Criteria**:
   - [ ] Bundle records define minimum metadata, ownership, state, and promotion rules
   - [ ] Promotion into normal AgentX issues is supported
   - [ ] The design stays repo-local and additive

3. **Canonical portability contract**
   - **User Story**: As a maintainer, I want portable host surfaces generated from canonical AgentX assets so that packaging does not drift by host.
   - **Acceptance Criteria**:
   - [ ] Canonical source files are identified for generated host outputs
   - [ ] Product identity remains stable across generated packs
   - [ ] Drift can be validated automatically

#### Could Have (P2)

1. **Bounded parallel delivery pilot**
   - **User Story**: As a maintainer, I want eligible work to be decomposed into safe parallel units so that delivery can accelerate without reducing review quality.

2. **Skill packaging and review scorecards**
   - **User Story**: As a skill author, I want stronger scaffolding and adoption checks so that reusable workflow intelligence stays high quality.

#### Will Not Have (Out of Scope)

- Replacing the current AgentX lifecycle or issue model
- Removing current workflow surfaces before phase-1 replacements exist
- Unbounded autonomous parallel execution
- Hand-maintained portability forks that bypass canonical repo assets

- [ ] Deterministic artifact checks are used before broader model synthesis

- [ ] Fallback behavior exists when artifact state is incomplete or ambiguous
- [ ] Advisory-first outputs can later be hardened without redesign

### 5.3 Non-Functional Requirements

#### Performance
- Workflow guidance must remain lightweight enough for normal operator use
- New product surfaces should prefer deterministic state resolution where possible

#### Security
- Reuse existing AgentX command, artifact, and boundary rules
- Do not create host-specific portability flows that bypass the repo's normal validation expectations

#### Scalability
- The design must work in both GitHub-backed and local artifact-centric workflows
- Portability generation must scale through canonical assets rather than manual duplication

#### Reliability
- Operators must still get clear guidance when not all artifacts are present
- Rollout scorecards and pilot sequencing must remain useful even if later workstreams are deferred

## 7. User Flows

### Flow 1: Guided Loop Entry
1. Operator starts from an issue, plan, or workflow command.
2. AgentX identifies the current checkpoint and available artifacts.
3. AgentX recommends the next action: deepen plan, start work, kick off review, or compound capture.
4. Operator takes the recommended action from chat, command, or sidebar surface.

### Flow 2: Phase-1 Rollout Control
1. Maintainer selects the next pilot slice.
2. Rollout scorecard defines success and rollback thresholds.
3. Exit criteria are checked before the next slice begins.
4. Release enablement guidance is updated for operators.

### Flow 3: Later-Phase Expansion
1. Guided loop contracts stabilize.
2. Decomposition and task-bundle rules are introduced.
3. Bounded parallel delivery pilots against eligible work.
4. Portability generation expands from canonical assets after workflow surfaces are stable.

## 9. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| New workflow language adds complexity instead of clarity | Operators see more steps but not better flow | Start with explicit checkpoint naming and next-step recommendations only |
| Rollout expands too quickly | Review quality and operator trust drop | Require scorecard thresholds and pilot exit criteria before broader expansion |
| Later decomposition work becomes a second backlog | Tracking drift and duplication | Require promotion into the normal AgentX issue hierarchy |
| Portability work creates host drift | Generated surfaces become inconsistent | Generate from canonical assets and validate output parity |
