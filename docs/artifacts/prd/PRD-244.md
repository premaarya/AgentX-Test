## 1. Problem Statement

### What problem are we solving?

AgentX already has durable plans, progress logs, checkpoint guidance, and loop controls, but long-running work still relies too heavily on broad phase transitions rather than bounded implementation contracts. As a result, complex work can drift between planning intent, implementation scope, evaluator feedback, and runtime proof.

### Why is this important?

AgentX is explicitly positioned as an autonomous long-running work system. If it cannot reliably turn complex work into bounded, testable, resumable slices with clear evaluation feedback, the product loses trust exactly where its value proposition is strongest.

### What happens if we do not solve this?

- Complex tasks remain harder to resume across sessions.
- Review and quality evidence stays stronger at the end than during `Work`.
- The workflow docs continue to describe evidence-backed behavior more strongly than runtime surfaces enforce it.
- Future model improvements are harder to exploit because the harness lacks an explicit policy for when to simplify or reset.

## 2. Target Users

### Primary Users

**User Persona 1: AgentX Operator**
- **Demographics**: Technical individual contributor or tech lead using VS Code and repo-local artifacts.
- **Goals**: Drive complex work with predictable status, evidence, and recovery paths.
- **Pain Points**: Long tasks become opaque; it is hard to tell what the next bounded step is and what has actually been proven.
- **Behaviors**: Uses execution plans, progress logs, commands, sidebars, and review artifacts during active work.

**User Persona 2: AgentX Engineer / Reviewer**
- **Demographics**: Developer or reviewer responsible for implementation quality.
- **Goals**: Receive scoped tasks with explicit proof requirements and actionable findings.
- **Pain Points**: Review feedback can arrive too late; implementation proof is inconsistent across complex work.
- **Behaviors**: Works from issue context, specs, execution artifacts, tests, and review findings.

### Secondary Users

- Product and architecture collaborators who need a trustworthy way to see whether long-running work is converging.
- Future plugin or workflow authors who need stable harness contracts instead of prompt-only conventions.

## 4. Research Summary

### Sources Consulted

- Anthropic engineering article `Harness design for long-running application development`.
- `docs/WORKFLOW.md`
- `AGENTS.md`
- `vscode-extension/src/utils/workflowGuidance.ts`
- `vscode-extension/src/utils/harnessState.ts`
- `.agentx/agentic-runner.ps1`
- Existing execution plans under `docs/execution/plans/`

### Key Findings

- Strong harnesses separate planning, implementation, and evaluation concerns rather than relying only on a single agent's self-assessment.
- Bounded contracts for each slice are more operationally useful than broad phase labels alone.
- Durable artifacts matter more than transcript memory for resumption and verification.
- Runtime verification on the real surface is more trustworthy than narrative assertions.
- Harness controls should remain only while they provide measurable lift; dead scaffolding should be removed.

### Chosen Approach Rationale

AgentX should extend the existing checkpoint model rather than replace it. The highest-value adoption is to make `Work` contract-driven, introduce adaptive evaluator participation for complex slices, and strengthen evidence expectations while preserving existing issue, plan, progress, and review artifacts.

### Rejected Alternatives

- Add a second dedicated harness lifecycle alongside checkpoints: rejected because it would split operator vocabulary and durable state.
- Make evaluator participation mandatory on all tasks: rejected because it would overburden simple bug/docs work.
- Treat the initiative as extension-only UI work: rejected because the contract spans docs, runtime, extension, and review surfaces.

## 5. Requirements

### 5.1 Functional Requirements

#### Must Have (P0)

1. **Contract-driven work slices**: AgentX must support a durable contract for bounded work inside the existing `Work` checkpoint.
 - **User Story**: As an operator, I want each complex slice to declare scope, acceptance, and proof requirements so that implementation stays bounded and resumable.
 - **Acceptance Criteria**:
 - [ ] A durable artifact shape exists for bounded work contracts.
 - [ ] The contract declares scope, exclusions, acceptance criteria, verification method, and recovery notes.
 - [ ] The workflow documentation describes how contract state fits into the existing checkpoint model.

2. **Adaptive evaluator participation**: AgentX must support evaluator involvement during `Work` for complex or high-risk slices.
 - **User Story**: As an engineer or reviewer, I want evaluator feedback before final review so that defects are surfaced earlier and findings are more actionable.
 - **Acceptance Criteria**:
 - [ ] The design defines when evaluator participation is optional, recommended, or required.
 - [ ] Evaluator findings can be attached to a slice and fed back into the next iteration.
 - [ ] The system does not require evaluator loops for every small task.

3. **Stronger runtime evidence model**: AgentX must distinguish implementation, verification, and runtime evidence for complex slices.
 - **User Story**: As a reviewer, I want concrete evidence expectations for each slice so that final approval is based on observable proof rather than narrative claims.
 - **Acceptance Criteria**:
 - [ ] The architecture defines evidence classes and where they live.
 - [ ] Target surfaces can summarize slice evidence durably.
 - [ ] Runtime verification is explicitly addressed for applicable work.

#### Should Have (P1)

1. **Context reset vs compaction policy**: AgentX should define when long-running work continues in-place, compacts context, or performs a clean-slate reset with durable handoff.
 - **User Story**: As an operator, I want the harness to choose between continuity and reset intentionally so that long tasks remain coherent.

2. **Contract-aware workflow guidance**: AgentX should surface current slice state, next action, and blockers in extension guidance and command surfaces.
 - **User Story**: As an operator, I want the system to show me the current contract state so that I do not have to infer it manually.

#### Could Have (P2)

1. **Pilot scorecard and pruning guidance**: AgentX could include a formal rubric for deciding which harness controls remain load-bearing after pilots.
 - **User Story**: As a maintainer, I want a simplification rubric so that obsolete scaffolding is removed deliberately.

#### Won't Have (Out of Scope)

- Full implementation of all runtime and UI surfaces in this planning phase.
- A second issue/status state machine separate from checkpoints.
- Mandatory evaluator loops for all tasks regardless of size or risk.

### 5.3 Non-Functional Requirements

#### Performance
- Workflow guidance should remain fast enough for interactive extension surfaces.
- Contract and evidence artifacts must remain lightweight and diff-friendly.

#### Security
- New harness artifacts must not require hardcoded credentials.
- Evidence links and findings must avoid exposing secrets in repo-local summaries.

#### Scalability
- The contract model must support multiple slices per complex task.
- The model must support future provider/runtime variation without redefining the workflow vocabulary.

#### Usability
- Operators should not need to learn a second workflow state machine.
- Child artifacts should be discoverable from existing docs, commands, and sidebars.

## 7. User Flows

### Primary Flow: Complex Work Slice

1. Operator starts from an execution plan under `Plan`.
2. AgentX enters `Work` and proposes a bounded contract for the next slice.
3. Evaluator participation is selected based on complexity and risk.
4. Implementation proceeds against the approved slice.
5. Verification and runtime evidence are captured.
6. Evaluator findings either approve the slice or return concrete adjustments.
7. The next slice begins or the task advances toward `Review`.

### Secondary Flow: Clean Reset

1. Long-running work loses coherence or exceeds safe context confidence.
2. AgentX writes or updates durable handoff state.
3. The next run resumes from plan, contract, findings, and evidence rather than chat memory.

## 9. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Contract model becomes too heavy for routine work | High | Keep evaluator and contract depth adaptive by task complexity |
| A second lifecycle emerges accidentally | High | Reuse checkpoint vocabulary and keep contract state nested inside `Work` |
| Docs get ahead of runtime again | High | Stage rollout and only claim enforcement where runtime supports it |
| Operators ignore the new artifacts | Medium | Expose contract state through existing commands and sidebars |
| Model capability changes make controls obsolete | Medium | Add explicit pruning guidance after pilot work |
